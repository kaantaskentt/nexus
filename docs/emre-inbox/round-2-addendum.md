1. The bug cluster resolved, and the lesson it leaves
The July 10 failures (snapshot never composing, plan drafting hanging, paste-compile erroring) were API token exhaustion. Once topped up, the queued work all completed on its own. Two takeaways:
Credit exhaustion must be a loud, named error on every AI surface. The system failed silently in three different costumes; nobody should spend an evening diagnosing an empty tank.
The queued retries all executed after top-up: Test Mest jumped from 57 records to 143 before either worker interview, suggesting repeated compile runs may have duplicated records. Worth a dedupe check on any workspace that hit the exhaustion window.

4. The “stupid questions” diagnosis (Ahmet session)
Emre’s field note: a real 9-5 respondent cannot be asked to sit through this many granular questions. The respondent snapped (“stupid questions,” “are you not listening”). Rediagnosed with Berk as control, three causes in order:
Stale definition-of-done. The retired data-cleaning entry demands “input, steps, tool, performer, time estimate, redo trigger.” The annoying questions map one-to-one onto those fields, including the twice-asked “rough feel or timed?”. The interviewer was dutifully filling a requirements list that was supposed to be retired (see bug 2 above).
Wrong register pre-read. The handling notes carried the sponsor’s “pleaser” characterization and the operator’s wrong role (office assistant). The pleaser playbook prescribes concrete-object anchoring and relentless episode probes; applied to an irritable, competent data scientist it reads as condescension. Pre-reads should be hypotheses the live conversation can override.
Residual polish: never repeat a source-probe in identical words; treat respondent irritation as a register signal.
Control condition: Berk’s session (clean mission, correct role, fresh definition of done) produced zero object-level misfires from the same interviewer on the same night.
