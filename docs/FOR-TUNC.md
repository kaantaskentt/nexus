# For Tunç — how this build relates to yours

First, the honest frame: what you built solo, at that pace, is impressive — the pipeline shape matches the methodology, and most of your architecture calls were right. We kept your skeleton on purpose: FastAPI + Postgres + pgvector, the in-database queue with SKIP LOCKED, the agent-run audit trail, prompt versioning. Those survive here because they were good decisions, and they're vendored with attribution.

This document is the running log of where we bent the design and why, so you can picture the delta quickly and adapt whatever you find useful. None of it is carved in stone — these are opinions held with reasons, and reasons can lose to better data. Where your instinct was scale-ready and ours is stage-appropriate, we've said so; both are defensible, we just optimized for the three clients in front of us.

| # | What changed | The thinking | If you want to adapt it |
|---|---|---|---|
| 1 | Strong model (Claude Sonnet-class+) now sits in the compiler/interviewer/planner seats; small models handle chunking and mechanical scoring | The runtime model was likely never an explicit decision — a default that survived. Moving it explained most of the perceived quality gap in one change, which is good news: the chassis was never the problem | Your `agent_configs` table already supports this — it's a data update, no deploy |
| 2 | Findings grew into typed claim records: `kind` (statement / directive / admission / correction), `topic` (8 kinds), trust `tag` (GUESS / CLAIMED / CONFIRMED), quarantine flags, `supersedes_id` | Typing at extraction decides what gets scored, committed, or discarded — one change at the root instead of stacked filters downstream. It also gives corrections real semantics (the 40→10 retraction case) | The full ontology with worked examples is in `prompts/agents/stage4-compiler.md` — it's written to be buildable-from directly |
| 3 | Pinecone dependency retired | Your own architecture brief makes the pgvector case well (atomic KB commits, metadata-first filtering) — the code just hadn't caught up with the doc yet | Drop `PineconeServices.py` + the requirement; the brief's migration-path note (company_id partitioning) is preserved |
| 4 | Consolidated storage/analytics onto Supabase; GCP, Firebase, S3, and ClickHouse rest for now | Nine vendors is a lot of keys, consoles, and explaining-to-clients for a three-client stage. Your setup reads as built-for-scale — that day may come, and this schema ports back without redesign | Nothing to do unless you want the simpler ops story too |
| 5 | "Candidate" split into three types: automation opportunity / knowledge gap / website suggestion | The single bucket made scoring incoherent (website copy carrying pain scores) and confused first users. Once split, the names name themselves | Rename follows the split naturally |
| 6 | Generation steps that claim KB grounding now require non-empty `retrieval_queries` | Keeps the audit trail honest — a grounded claim with an empty retrieval record is the one combination we can't allow | Small hard check in the agent runner |
| 7 | Voice (VAPI) plugged into `run_interview_turn` | Genuine credit here: your turn engine was transport-agnostic without needing to be — which made voice an entrance, not a rewrite | The webhook adapter pattern will be in `backend/` once built |
| 8 | Invite tokens got expiry + session binding; "Mark Complete" shortcut removed; plans can never request credentials | Standard hardening pass, worth porting as-is | Small patches, self-contained |
| 9 | Client-facing surfaces (activity feed, provenance views, the compiled report) no longer render internals — raw errors, temp passwords, and JSON dumps moved behind an internal flag; the report itself is getting a structural redesign | Kaan and Emre flagged this together: for a product whose pitch is trustworthy capture, the client-visible layer *is* the product. The report as it stood read as a data dump rather than a deliverable — the new Stage 8 design (workflow canvas + findings + gaps) replaces it | The mockups in `reference/drive-stage-docs/` show the target |

| 10 | Email provider deferred — invite sending sits behind a single `send_invite()` interface with a console/log stub for now | Twilio/SendGrid credential setup was friction the demo doesn't need; the invite renders in-app either way. Provider (Resend or your SendGrid) plugs into one file later | Your SendGrid wrapper slots straight in when the time comes |

| 11 | `claim_records.tag` is nullable; only STATEMENT/CORRECTION carry a trust tag, DIRECTIVE/ADMISSION carry null | The trust ladder (GUESS→CONFIRMED) only makes sense for factual claims. A directive ("don't mention Harrods") or an admission ("I don't know how it works") isn't more-or-less trustworthy — forcing a tag would pollute client trust displays | Migration `0002_ontology_ops.sql`; render logic filters by `kind` for the NEVER list vs. learned cards |
| 12 | The compiler persists in a two-pass insert: records get stable local ids (`r1`, `r2`) at generation, superseded links are wired after all rows exist; sentiment quarantine is a DB trigger, not prompt discipline | Within-call corrections (40→10) must supersede a record minted in the same batch, before any UUID exists. And "one leaked judgment kills the product" is too important to leave to the model behaving — the trigger force-locks any `sentiment_flag` row regardless of what the LLM returned | `backend/app/pipeline/compiler.py` + the `claim_records_quarantine_sentiment` trigger in 0002 |
| 13 | Four compounding learning loops now ride on your instrumentation: per-industry heuristic-accuracy scoring, per-client record compounding (cross-interview memory), failure-mined evals (the failure taxonomy → regression tests), and versioned prompt evolution | This one is pure credit: none of the four are possible without the two tables you built — `agent_runs` (every call's model, `prompt_version`, `retrieval_queries`, tokens, latency, and output) and prompt versioning. Because every run is audited and every prompt carries a version, we can measure what actually moved quality, attribute a gain to a specific prompt version, and evolve on evidence instead of vibes. That turns "the system gets better over time" from a slogan into something the schema can actually prove | Nothing to adapt — it's your audit substrate used for exactly what it was built for; the loops read from `agent_runs` joined against the prompt-version history |

| 14 | Snapshot drawer's pain-signal stat boxes dropped the mini progress bar the V2 design spec called for | The bar rendered a fixed two-thirds fill for every signal — an invented proportion, since the values are qualitative ("Daily, every morning", "Raised as a primary pain point"), not scores. A meter that implies a measurement we don't have violates the honesty rule (F28/A2), so the elevated stat box now lets the value carry the signal | Design-V2 §4.4 note; if a real 0–N signal ever exists, a token-driven bar can return honestly |

New entries get added below as the build proceeds — same format, short on purpose. If you disagree with any call here, say so; changed minds are cheaper than wrong architecture.
