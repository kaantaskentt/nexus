# For Tunç — how this build relates to yours

First, the honest frame: what you built solo, at that pace, is impressive — the pipeline shape matches the methodology, and most of your architecture calls were right. We kept your skeleton on purpose: FastAPI + Postgres + pgvector, the in-database queue with SKIP LOCKED, the agent-run audit trail, prompt versioning. Those survive here because they were good decisions, and they're vendored with attribution.

This document is the running log of where we bent the design and why, so you can picture the delta quickly and adapt whatever you find useful. None of it is carved in stone — these are opinions held with reasons, and reasons can lose to better data. Where your instinct was scale-ready and ours is stage-appropriate, we've said so; both are defensible, we just optimized for the three clients in front of us.

| 13 | Four compounding learning loops now ride on your instrumentation: per-industry heuristic-accuracy scoring, per-client record compounding (cross-interview memory), failure-mined evals (the failure taxonomy → regression tests), and versioned prompt evolution | This one is pure credit: none of the four are possible without the two tables you built — `agent_runs` (every call's model, `prompt_version`, `retrieval_queries`, tokens, latency, and output) and prompt versioning. | 18 | Computed coverage-routing was built (new `coverage_tracker` agent seat + `coverage.py`: server-side satisfied/partial/untouched objective map, close-gated on untouched must-hits) but ships defaulted OFF behind `coverage_routing` | The turn engine re-derives objective coverage in the interviewer's head; the V3 proposal was to compute it server-side and force a probe to any untouched must-hit before close. We built it and A/B'd it on the real engine. The finding: with an untouched item made an explicit must-hit OBJECTIVE, the baseline interviewer already routes to it and covers it (3/3 x2 runs); the computed-coverage version added a ~3.4s/turn model call for no measured gain (and was noisier). The motivating misses (h-bk-3, ag-2) are HIDDEN knowledge, not objectives, so an objective-coverage gate can't force them to surface. | 19 | Pipeline agent-JSON calls no longer swallow parse failures. The `extract_json(run_agent(...)) except ValueError: return` idiom in six fan-out legs (quality, conflicts×2, heuristics×2, recon, workflow, snapshot) is replaced by a single `run_agent_json()` helper that logs at ERROR and raises `AgentParseError` — so the owning job fails and retries via your SKIP-LOCKED queue instead of reporting `done` with nothing written. The `agent_runs` audit now persists the raw output text (`output_ref.text`), not just its char count | A verbose agent (interview_quality: 7329 chars; perception_gap: 1559 tokens) produced complete output that `extract_json` couldn't parse; the leg swallowed it and the job still went `done`, so `interview_quality` and `perception_gaps[]` came out empty with no error anywhere — the report looked complete but wasn't. A silent drop at the payoff moment is exactly the failure your audit trail exists to prevent; this makes it loud, retryable, and debuggable from the audit row | `backend/app/llm.py` (`run_agent_json` + `AgentParseError` + the `output_ref` text persist); the queue's existing failed/retryable handling does the rest. The agent prompts that emit unparseable verbose output are constrained separately (task #23) — this change is the safety net, not the cure |

| 20 | The confidence/trust pill can now carry an opt-in `context="Trust"` prefix (a low-emphasis "Trust \| " segment); enabled only on the Snapshot learned cards, conflict cards, and area-drawer beliefs. Colors, labels, and the F35/A2 tier mapping are untouched | A taste audit flagged that the green "High" tier pill, sitting next to a bad-news headline ("account nearly lost"), can misread as "this is good" rather than "high trust". Relabeling (not recoloring) disambiguates without touching the locked semantics. Kept off by default so trust-native surfaces (Insights, Knowledge) are byte-identical | `ConfidenceBadge` optional `context` prop; **flagged for Kaan's veto** since it touches the client-facing trust surface. Revert = drop the three `context="Trust"` props |

| 21 | The API is no longer open (closes the entry-16 posture). A FastAPI dependency (`backend/app/auth.py:require_admin`) now verifies the caller's Supabase JWT against GoTrue (`GET /auth/v1/user`) and gates every admin router in `main.py`; the interviewee `by-token/*` routes and the secret-gated voice callbacks stay public by design. The browser sends the token (frontend `api.ts` + a server/client token split), and the eval harness authenticates for real via a GoTrue password grant (`evals/harness/auth.py`) — no localhost allow-list | The YC eng audit rated the open API the one P0 that would bite a client: a workspace UUID was the only credential, and `plans/send` (which mints interviewee tokens = the "talks to employees" boundary) was reachable unauthenticated. Entry 16 already named this the real boundary and the exact fix; this is that fix landing. GoTrue verification (not local secret decode) was chosen so it works for both HS256 and asymmetric signing keys and needs no secret the backend didn't already hold | New env the deploy must set: backend `SUPABASE_ANON_KEY` (verification apikey); harness `NEXUS_ADMIN_EMAIL` / `NEXUS_ADMIN_PASSWORD` (never committed). Tenant-scoping-by-claims is still open — this proves identity, not per-workspace authorization; that's the next layer |
